# ShubhamGPT

ShubhamGPT is a full-stack chat application inspired by ChatGPT. The frontend is built with React and Vite, and the backend is built with Express, MongoDB/Mongoose, and the Gemini API. Users can start a new chat, send prompts, receive AI replies, view previous chat threads, switch between threads, and delete old threads.

## Project Structure

```text
ShubhamGPT/
  Backend/
    models/
      Thread.js          # Mongoose schema for chat threads and messages
    routes/
      chat.js            # API routes for chat, thread list, thread details, delete
    utils/
      openai.js          # AI API helper. Currently calls Gemini API
    init.js              # Optional seed script for sample thread data
    server.js            # Express server and MongoDB connection
    package.json
    .env                 # Local secrets, ignored by Git

  Frontend/
    src/
      App.jsx            # Global state provider and main layout
      MyContext.jsx      # React context shared by sidebar and chat window
      Sidebar.jsx        # New chat, thread history, thread delete
      ChatWindow.jsx     # Input box, request handling, loading state
      Chat.jsx           # Chat message rendering and typing effect
      *.css              # Component styling
    package.json

  README.md
  .gitignore
```

## Tech Stack

- Frontend: React 19, Vite, CSS
- Markdown rendering: `react-markdown`, `rehype-highlight`
- Loading UI: `react-spinners`
- ID generation: `uuid`
- Backend: Node.js, Express
- Database: MongoDB with Mongoose
- AI provider: Gemini API
- Environment variables: `dotenv`

## How The App Works

1. The user opens the React app from the frontend.
2. `App.jsx` creates shared state using `MyContext`.
3. `Sidebar.jsx` fetches all saved threads from `GET /api/thread`.
4. When the user sends a message, `ChatWindow.jsx` posts the message and current `threadId` to `POST /api/chat`.
5. The backend checks MongoDB for that thread.
6. If the thread does not exist, the backend creates a new thread with the first user message as the title.
7. If the thread exists, the backend appends the new user message.
8. `utils/openai.js` sends the prompt to Gemini and returns the AI response.
9. The backend saves the assistant reply in MongoDB.
10. The frontend receives the reply, updates local chat state, and `Chat.jsx` renders it with a typing effect.

## Backend API

The backend runs on:

```text
http://localhost:5000
```

Routes:

```text
GET    /api/thread              # Get all chat threads, newest first
GET    /api/thread/:threadId    # Get messages for one thread
POST   /api/chat                # Send a message and get AI reply
DELETE /api/thread/:threadId    # Delete one thread
POST   /api/test                # Test route for saving sample data
```

`POST /api/chat` expects:

```json
{
  "threadId": "unique-thread-id",
  "message": "Your prompt here"
}
```

It returns:

```json
{
  "reply": "AI response here"
}
```

## Database Model

MongoDB stores chat threads using the `Thread` model.

Each thread has:

- `threadId`: unique ID generated by the frontend
- `title`: first user message, used in the sidebar history
- `messages`: array of user and assistant messages
- `createdAt`: when the thread was created
- `updatedAt`: when the thread was last updated

Each message has:

- `role`: either `user` or `assistant`
- `content`: message text
- `timestamp`: message creation time

## Environment Variables

Create this file:

```text
Backend/.env
```

Add:

```env
MONGODB_URI=your_mongodb_connection_string
GEMINI_API_KEY=your_gemini_api_key
GEMINI_MODEL=gemini-2.5-flash-lite
```

Notes:

- `MONGODB_URI` is required for saving and loading chats.
- `GEMINI_API_KEY` is required for AI replies.
- `GEMINI_MODEL` is optional. If missing, the backend uses `gemini-2.5-flash-lite`.
- Do not push `.env` to GitHub. It is ignored by `.gitignore`.

## Frontend State Flow

`App.jsx` owns the main app state:

- `prompt`: current input text
- `reply`: latest AI reply
- `currThreadId`: active chat thread ID
- `prevChats`: messages in the current thread
- `newChat`: whether the current screen is a fresh chat
- `allThreads`: sidebar thread list

These values are shared through `MyContext`.

`Sidebar.jsx`:

- Loads all threads
- Starts a new chat
- Opens an old chat
- Deletes a thread

`ChatWindow.jsx`:

- Reads the input prompt
- Sends `threadId` and `message` to the backend
- Shows loading spinner
- Stores the latest reply

`Chat.jsx`:

- Displays user and assistant messages
- Renders markdown responses
- Highlights code blocks
- Shows the latest reply word by word

## Important Implementation Notes

- The frontend currently uses hard-coded API URLs like `http://localhost:5000/api/chat`.
- The backend port is hard-coded as `5000` in `Backend/server.js`.
- The helper file is named `Backend/utils/openai.js`, but the active code calls the Gemini API.
- There is commented OpenAI code in `utils/openai.js` and `server.js`; the current working provider is Gemini.
- Chat history is saved in MongoDB, not browser local storage.
- Thread titles are currently the first user message.

## Common Problems

Backend says `GEMINI_API_KEY is missing in .env`:

- Check that `Backend/.env` exists.
- Check that the variable name is exactly `GEMINI_API_KEY`.
- Restart the backend after editing `.env`.

Backend cannot connect to database:

- Check `MONGODB_URI`.
- Make sure the MongoDB Atlas IP access list allows your current IP.
- Make sure the database username and password are correct.

Frontend does not get replies:

- Make sure backend is running on port `5000`.
- Open browser dev tools and check the console/network tab.
- Confirm `POST http://localhost:5000/api/chat` is not failing.

Old chats do not show:

- Make sure MongoDB is connected.
- Check `GET http://localhost:5000/api/thread`.
- Make sure the current database has thread documents.
